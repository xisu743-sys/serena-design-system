01 · before

手动刷的那段时间，问题其实很清楚。

为了找 YouTube 频道定位，我空下来就刷视频，每天两三个小时。 最想捕捉的是低粉异常爆款，但现有工具不是太贵，就是只把数据被动放在侧边栏。

01

看不到增量

插件只显示发布时间和当前播放量，无法判断最近 24 小时到底有没有爆发。

02

付费门槛高

vidIQ、ViewStats 信息零散，前期不想为一个还没验证的需求持续订阅。

03

没有主动提醒

工具都在等我打开页面。爆款发生时，它不会主动推门进来告诉我。

04

低粉爆款被淹没

5k-10k 小频道的异常增长最有参考价值，但它们最容易被大频道内容盖过去。

02 · build

11 天，158 次提交，一条链路跑通了。

这个项目最有意思的地方不是功能列表，而是每一步都来自真实使用里的摩擦： 想看增量、想要通知、想沉淀样本、想在刷 YouTube 的时候直接加频道。

1

先把数据拉回来

接入 YouTube Data API，拉取频道和视频基础数据。MVP 雏形先跑起来，再用真实数据校准需求。

2

增量计算绕了弯路

一开始从发布时间起算，结果数值固定，完全不能判断最近有没有爆。后来改成以当前时间为基点倒推 24h / 7d。

commit: 增量改为滚动倒推窗口

定时任务第一天没动静

03 · hard parts

真正卡住的，不是写页面，是把判断逻辑写对。

growth-window.ts 
// 正确：滚动倒推窗口，而不是从发布时间累计
const cutoff = Date.now() - 24 * HOUR;
const baseline = await findSnapshotBefore(videoId, cutoff);
const delta = current.views - baseline.views;

if (delta > rule.threshold) {
  await sendNotification('企业微信', video);
}
note one

滚动窗口，不是累计

从发布时间起算只能得到一个静态差值。真正要看的，是“现在往前推 24 小时”的增长。

note two

endpoint 错误不会替你尖叫

QStash 第一天完全静默，最后发现是 regional endpoint 配错。生产环境里，沉默也是一种报错。

note three

配额优化就是产品能力

如果一次同步消耗太高，功能就无法常态运行。把 100 单位降到 1 单位，本质上是在扩大产品边界。

第一次收到爆款通知时，我知道这个工具的闭环成立了。
这次跑在生产环境里的，不是 demo。

rules triggered · wecom notification · real signal delivered

06 · next

想继续做的事。

near term

让“机会挖掘”真正上线

  • 短视频和长视频分开展示，避免信号混在一起。
  • 在对标组内加入拆解记录：标题、开头、情绪钩子、叙事结构。
  • 更精细的阈值设置，目标锁定千万级内容信号。
further

从“已知频道监控”到“全网发现”

  • 自动检索 5k-10k 订阅小频道的异常增长。
  • 支持多用户与并发处理，让监控池真正扩大。
  • 加入 AI 辅助趋势判断，但不把判断权完全交出去。

当前版本做到的是：把已知对标频道的监控、判断、通知、样本沉淀串成一条链。“全网低粉爆款发现”是产品目标，但前一条链跑通了，才知道下一条要怎么搭。
